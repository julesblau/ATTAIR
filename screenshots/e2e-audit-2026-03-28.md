# ATTAIRE UI Quality Audit — 2026-03-28

**Auditor:** Design Critic / QA Agent (Opus 4.6)
**Device:** iPhone 14 Pro (390x844 @ 3x)
**Mode:** Headless Playwright, dark + light mode
**Backend:** Unavailable (no local .env) — tested UI shell, empty states, and all reachable screens

---

## Executive Summary

ATTAIRE has a **strong design foundation**: the gold accent (#C9A96E), dark-mode-first approach, Outfit font, and overall brand identity are premium and distinctive. The onboarding and auth flows are polished and would compete with top-tier apps. However, multiple structural UI issues prevent a PASS verdict — particularly a **ghost search bar** bleeding through on every tab, a **dead History tab** with no entry point, and the **Saved tab failing to render** its own content. These are not cosmetic — they are core navigation failures that would confuse users on first use.

**Overall Verdict: NEEDS WORK**

### 1. Onboarding (01-onboarding-step1.png)

**Verdict: PASS**

This is strong. The first screen a new user sees nails the luxury fashion app feel.

**What works:**

The gender preference, budget range, and size preference steps still exist in the JSX rendering code (lines 3025-3069) and the step types (budget_range, size_prefs, select) but are never reached because `OB_STEPS` at line 1081 only defines 2 entries. - **(LOW)** Step 2 ("Scan Your First Outfit") shows "Take a Photo" and "Upload" buttons, but both just redirect to the signup page — they don't actually let you scan. This is mildly deceptive. The user expects to scan but gets a signup form. Consider changing the CTA to "Create Account to Scan" to set expectations.

### 4. Home Feed — Dark Mode (06-home-dark.png)

**Verdict: NEEDS WORK**

The empty state is clean but reveals several issues.

**What works:**
- Header with "ATTAIRE" logo in Instrument Serif italic is distinctive and premium
- "FREE - 12/12" badge is informative and tappable (leads to upgrade modal)
- "For You" / "Following" tabs are clean Instagram-style toggles
- Empty state copy is encouraging ("Scan your first outfit to discover styles")
- "Scan an Outfit" CTA button is centered and prominent
- FAB at bottom with camera icon and gold background is clearly a scan trigger
- Bottom nav with 4 icon-only tabs is minimal and clean

**Issues:**
- **(HIGH)** **Bottom nav has NO text labels.** The 4 bottom nav buttons (Home, Search, Saved, Profile) are icon-only with no text underneath. Compare to Instagram, TikTok, and GOAT — all have text labels below icons. Without labels, users must guess what each icon means. The heart icon could be "likes" or "saved" or "notifications." The magnifying glass could be "search" or "explore." This violates iOS Human Interface Guidelines which recommend text labels for tab bars. The `.tab-l` CSS class for labels exists (line 2397) but the bottom bar at lines 5577-5598 never renders any `<span>` text labels.
  - **Suggested fix:** Add `<span style={{ fontSize: 10, marginTop: 2 }}>Home</span>` etc. under each SVG icon in the bottom tab bar.
- **(HIGH)** **Ghost search bar visible at bottom of screen.** In the search tab screenshot (08-search-tab.png) and the saved tab screenshot (11-saved-dark.png), a semi-transparent "Search people... Ca..." text is visible at the very bottom of the screen, overlapping with the FAB and tab bar. This appears to be the `user-search-overlay` (line 5492) leaking through. The overlay is `position:fixed;inset:0;z-index:200` but the tab bar is `z-index:1000` and the FAB is `z-index:1001`, so the overlay's input/cancel buttons get stuck behind them but are still partially visible.
  - **Root cause:** When switching to the Search tab, `showUserSearch` gets set to `true`. The inline search bar in the Search tab content (line 3378) and the full-screen overlay (line 5492) both render simultaneously, creating a double-search-input situation. The overlay's bottom area peeks through behind the tab bar.
  - **Suggested fix:** When `tab === "search"`, don't show the separate `user-search-overlay`. The Search tab already has its own inline search input. Remove the overlay trigger from the Search tab or gate it with `tab !== "search"`.
- **(MEDIUM)** **Saved tab shows Home feed content.** The 11-saved-dark.png screenshot shows "For You" / "Following" tabs and "Scan your first outfit" — this is the Home feed, not the Saved tab. The Saved tab's `{tab === "likes" && ...}` block (line 4697) should render its own header ("Saved") and empty state. This suggests the tab switch didn't work, or the Home feed content is always visible. Looking at the code: the Home feed renders when `tab === "home"`, the Likes tab when `tab === "likes"`. The screenshot appears to be taken when the tab was still "home" despite clicking the Saved button. This may be a timing issue in the test OR a real bug where the Saved button's click handler fails silently.
  - **Action:** Verify by manually testing in a browser. If the tab click works, this is a test timing issue. If it doesn't, investigate the click handler.

### 6. Search Tab (08-search-tab.png)

**Verdict: FAIL**

This screen has multiple overlapping elements and a confusing UX.

**What works:**
- "Search people..." placeholder is clear
- Magnifying glass icon is correctly placed
- Empty state ("Search for people to follow") with subtitle is helpful
- The inline search bar at top of the Search tab content is well-designed

**Issues:**
- **(CRITICAL)** **Double search bar.** There is an inline search input in the Search tab content (line 3378, visible at top of screen) AND a second search input from the `user-search-overlay` (line 5494, visible at the very bottom, partially behind FAB). Users see two "Search people..." inputs simultaneously. This is confusing and broken.
- **(HIGH)** **FAB overlaps Cancel button.** The overlay's Cancel button at the bottom is directly behind the FAB (gold camera button). The Playwright test confirmed this — clicking Cancel times out because the FAB intercepts pointer events (logged in the first test run). Users cannot dismiss the search overlay by tapping Cancel.
- **(HIGH)** **"Type a name to search" text from overlay** appears at the bottom of the screen behind the tab bar, creating visual clutter.
- **(MEDIUM)** The search icon button in the header (line 3280) only shows on Home and Scan tabs (`tab === "home" || tab === "scan"`), which is correct — but when the user is already on the Search tab, the header still shows "ATTAIRE" and "FREE - 12/12" without a search icon. This is consistent but the lack of a back/close affordance is notable.

### 8. History Tab

**Verdict: FAIL**

**Issues:**
- **(CRITICAL)** **History tab is completely inaccessible.** The History tab rendering code exists at line 4252 (`tab === "history"`) with a full UI including scan cards, filters (All / Saved), detail overlays, rename, delete, and more. However, `setTab("history")` is **never called anywhere in the codebase**. There is no button, link, or navigation path that sets the tab to "history". The bottom nav bar has 4 tabs: Home, Search, Saved, Profile — no History.
  - This means the entire History feature (approximately 450 lines of JSX, lines 4252-4695) is dead code that users can never reach.
  - History was presumably a 5th tab that was removed when the nav was redesigned to the current 4-tab + FAB layout.
  - **Impact:** Users have no way to see their past scans. The Profile tab shows "0 Scans" and "Scan outfits to fill your grid" but the grid only shows scans from the profile's own history — there's no way to browse previous scans, re-open results, rename scans, or delete them.
  - **Suggested fix:** Either restore History as a tab, or integrate it into the Profile tab as a scrollable scan grid (Instagram-style).

## Cross-Cutting Issues

### A. Architecture / Code Quality

| Severity | Issue | Location |
|----------|-------|----------|
| **CRITICAL** | **6,080-line single-file App.jsx.** This monolithic file contains the entire app — all 15+ screens, all state, all styles, all API calls. This is unmaintainable. A single typo breaks everything. No code splitting means the entire bundle loads upfront. Instagram, TikTok, and GOAT all use component-based architectures with code splitting. | `src/App.jsx` |

## Priority Fix List (Recommended Order)

1. **Fix ghost search bar** — Gate the `user-search-overlay` to not render when `tab === "search"`. This fixes the double input and the FAB-blocking-Cancel issues simultaneously.
2. **Add text labels to bottom nav** — Under each SVG icon in the tab bar, add Home/Search/Saved/Profile labels.
3. **Restore or redirect History** — Either add History back as a tab, add it as a section in Profile, or add a "View History" button somewhere reachable.
4. **Fix phone input opacity** — Remove `opacity: 0.7` from the phone input on the signup form.
5. **Clean up dead code** — Remove the unreachable History tab JSX, the old onboarding steps, and the `{false && ...}` legacy profile.
6. **Light mode polish** — Improve button contrast, add warmth to profile, increase tab underline visibility.
